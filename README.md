# Minesweeper-based-SDL3

一个基于 `C++20`、`SDL3` 与 `SDL3_ttf` 的扫雷示例工程。

项目采用**状态机 + 逻辑渲染分层**的组织方式，将棋盘规则、状态切换与 SDL 渲染职责拆分，适合作为：

- `SDL3` 入门项目参考
- 小型游戏架构练习样例
- 扫雷玩法扩展与重构实验基础

## 项目简介

该项目围绕经典扫雷玩法实现了完整的基础流程，包括菜单、游戏中、暂停以及胜负状态切换。核心逻辑集中在无 SDL 依赖的 `Board` 中，方便后续做单元测试与规则验证；渲染层则统一由 `Renderer` 管理，配合脏格子更新减少重复绘制。

## 功能特性

- 使用 `C++20`
- 基于 `SDL3` 与 `SDL3_ttf`
- 状态机管理界面流程与状态切换
- `Board` 独立负责扫雷核心逻辑
- 逻辑层与渲染层分离，便于维护与扩展
- 使用脏格子思路减少无效重绘
- SDL 资源采用 RAII 管理
- 适合后续补充测试、资源缓存与 UI 优化

## 技术栈

- 语言：`C++20`
- 图形与输入：`SDL3`
- 文本渲染：`SDL3_ttf`
- 开发环境参考：`Visual Studio 2022`

## 项目结构

```text
Minesweeper_based_sdl3/
├─ README.md
└─ Minesweeper_based_sdl3/
   ├─ main.cpp
   ├─ Game.h / Game.cpp
   ├─ State.h / State.cpp
   ├─ StateMachine.h
   ├─ Board.h / Board.cpp
   ├─ Renderer.h / Renderer.cpp
   ├─ Constants.h
   └─ resource.h
```

各模块职责如下：

- `main.cpp`  
  程序入口，负责创建并启动 `Game`。

- `Game.h` / `Game.cpp`  
  负责游戏初始化、主循环、资源入口与整体生命周期管理。

- `State.h` / `State.cpp`  
  定义状态接口以及具体状态实现，例如菜单、游戏中、暂停、结束等。

- `StateMachine.h`  
  管理状态切换逻辑，并支持暂停时保留底层状态用于继续渲染。

- `Board.h` / `Board.cpp`  
  纯逻辑扫雷棋盘模块，负责布雷、翻开、插旗、洪泛展开、胜负判定等，不依赖 SDL，便于测试。

- `Renderer.h` / `Renderer.cpp`  
  负责窗口、渲染器、字体、纹理等 SDL 资源管理，并承担文本与棋盘绘制相关职责。

- `Constants.h`  
  定义格子尺寸、难度预设等全局常量。

- `resource.h`  
  存放资源相关定义或项目资源声明。

## 架构说明

### 1. 棋盘逻辑独立

`Board` 不直接依赖 SDL，仅处理规则与状态数据。这种设计带来的好处：

- 便于单元测试
- 降低渲染层耦合
- 后续更容易替换 UI 表现形式

### 2. 状态机驱动流程

项目通过状态机组织以下典型流程：

- 主菜单
- 游戏中
- 暂停
- 胜利 / 失败

这种方式比将所有逻辑堆叠在单一循环中更清晰，后续添加设置页、动画页或结算页也更方便。

### 3. 渲染职责集中

`Renderer` 统一管理 SDL 资源与绘制接口，避免纹理、字体、窗口句柄散落在不同模块中。

### 4. 脏格子更新

在游戏进行过程中，仅更新发生变化的格子区域，减少重复绘制，提高渲染效率。

## 运行方式

### 环境要求

- `Visual Studio 2022`
- 支持 `C++20` 的编译器
- `SDL3`
- `SDL3_ttf`

### 构建步骤

1. 克隆仓库

   ```bash
   git clone https://github.com/aldxc/Minesweeper_based_sdl3.git
   ```

2. 使用 Visual Studio 打开项目。

3. 在项目属性中配置 `SDL3` 与 `SDL3_ttf`：
   - 添加头文件目录（include）
   - 添加库目录（lib）
   - 配置对应链接库

4. 将 `SDL3` 与 `SDL3_ttf` 运行时所需的 DLL 放到可执行文件目录，或加入系统可搜索路径。

5. 选择 `Debug` 或 `Release` 配置后编译运行。

## 操作说明

- 鼠标左键：打开格子
- 鼠标右键：切换旗子标记
- 首次点击后触发布雷逻辑
- 在菜单中选择不同难度开始游戏

## 代码风格约定

- 类名使用 PascalCase，如 `Board`、`Game`、`Renderer`
- 成员函数与变量使用 camelCase，如 `resetBoard`、`placeMines`
- 头文件尽量使用前向声明，减少编译依赖
- 注释以中文为主，便于项目维护
- 随机数引擎作为类成员初始化，保证行为可控与可复现
- 创建 SDL 资源时进行有效性检查

## 当前设计亮点

- `Board` 完全脱离 SDL，适合作为独立逻辑模块维护
- `Renderer` 以 RAII 方式管理 SDL 资源
- `StateMachine` 有利于暂停界面与底层界面叠加渲染
- 项目结构清晰，便于继续扩展功能与 UI
